    # 什么是架构？

    实际上，架构是一种抽象能力。一般来说，架构师在代码能力达到某个阶段时，将升华为使用文档把架构蓝图展示出来，用以描述出抽象业务，并注入自身开发哲学。

    # 可能产生的误解

    在工作中，我观察到不少工程师所理解的架构是一套技术结构。其实我认为，在我们定义架构时，并不需要着重于“一套”、“技术”这样的观点上。

    其实架构也是经常调整的，在工作中我们会根据需求进行决策，需要和各个职能的人进行沟通。而技术，只是其中比较核心的部分，就如同我在 架构背景 中所描述的情况。与精妙高深的技术和技巧相比，我更偏向于如何解决当前问题、为未来提供更广阔的空间和对竞争对手产生技术优势。

    # 架构宣言

    这份文档的架构并不是面向高精尖的技术团队，相反我们需要考虑在大团队背景下，多维度的团队问题。简单的说法是：对初中级工程师友好。

    所以我们的架构宣言如下：

    # 提高简易性令技术更友好

    # 划分技术和人才水平令工程化更有效落实

    # 掌握关键核心令项目稳定性和安全性得到基本保障

    # 最终到达令团队产生规模效应的目的
